## 1. 제품 비전

**제품명**: **BuildWise** (buildwise.ai)

**한 줄 요약**: 자연어로 건물을 설명하면, AI가 3D 모델을 만들고, 에너지 시뮬레이션을 실행하고, 최적 절감 전략을 추천하는 SaaS 플랫폼.

### 핵심 가치 제안

| 기존 방식 | BuildWise |
|-----------|-----------|
| EnergyPlus IDF 수동 작성 (전문가 전용) | 자연어 입력 → 자동 생성 |
| Blender/SketchUp 별도 학습 필요 | AI가 3D 모델 생성, 웹에서 수정 |
| 시뮬레이션 결과 CSV/텍스트 분석 | 인터랙티브 대시보드 |
| EMS 전략 수동 적용/비교 | 원클릭 M0~M8 자동 비교 |
| 수주~수개월 소요 | 수분~수시간 |

### 경쟁 포지셔닝

```
               전문성 높음
                  │
    EnergyPlus    │   DesignBuilder
    (CLI/IDF)     │   (전문 GUI)
                  │
    ─────────────-┼──────────────── 자동화 높음
                  │
    OpenStudio    │   ★ BuildWise
    (SDK)         │   (NL → 자동 → 대시보드)
                  │
               전문성 낮음
```

## 2. 사용자 페르소나

### P1: 건물주 (비전문가)
- "내 건물 에너지 비용을 줄이고 싶다"
- 기대: 건물 사진/도면 업로드 → 절감 전략 추천 → ROI 확인
- 전문 용어 이해 불가, 시각적/직관적 결과 필요

### P2: 시설관리자 (준전문가)
- "HVAC 운영 조건을 바꾸면 얼마나 절약되나 비교하고 싶다"
- 기대: 운영 조건(스케줄, 설정온도) 조정 → 즉시 비교
- 기본 HVAC 용어 이해, 상세 설정 가능

### P3: 에너지 컨설턴트 (전문가)
- "고객에게 에너지 절감 보고서를 빠르게 제공하고 싶다"
- 기대: 정밀한 파라미터 제어, 커스텀 EMS, PDF 보고서 자동 생성
- IDF 직접 편집 가능, 고급 분석 필요

## 4. 모듈별 상세 스펙

### 4.1 모듈 ① — 건물 생성 (Building Creator)

#### 4.1.1 입력 방식 (4가지)

**A. 자연어 입력 (Primary — Phase 2)**

```
사용자: "서울 강남에 있는 12층 유리 커튼월 오피스 빌딩,
        지하 2층 주차장, 1층 로비, 연면적 약 46,000㎡"
```

**처리 방식 (확정: 자동채움 + 사후확인)**:
1. LLM이 자연어에서 건물 파라미터를 추출
2. 확실한 정보: 직접 매핑 (용도, 층수, 면적, 위치)
3. 추론 가능: AI가 합리적 기본값 적용 (기후대, HVAC, WWR)
4. 누락 정보: 건물유형+지역 기반 기본값 자동 적용
5. **사후 확인 카드** 표시 → 사용자가 수정 또는 승인

```
┌─────────────────────────────────────┐
│ 건물 정보 요약              [수정]   │
│                                     │
│ 🏢 대형 오피스 │ 서울 강남 │ 4A     │
│ 12층 │ 46,000㎡ │ VAV+Chiller      │
│                                     │
│ ⚠️ 다음 항목은 기본값입니다:        │
│ · 운영시간: 08:00~18:00 [변경]      │
│ · 창호비: 40% [변경]               │
│ · 냉방설정: 24°C [변경]            │
│                                     │
│ [확인하고 다음 단계로]              │
└─────────────────────────────────────┘
```

**설계 원칙**: 질문 최소화 → 이탈률 방지. "보여주고 수정하게" 패턴.

**B. 사진/도면 업로드 (Phase 3)**
```
입력: 건물 외관 사진, 평면도 CAD/PDF
처리: Vision AI → 건물 특성 추출 → 파라미터화 → 3D 생성
출력: 3D 모델 + 추출된 파라미터 (수정 가능)
```

**C. 기존 3D 모델 임포트 (Phase 3)**
```
지원 형식: .blend, .obj, .fbx, .ifc (BIM), .gbXML
처리: Blender headless → 형상 분석 → Zone 자동 분할 → 파라미터 추출
```

**D. 건물 유형 템플릿 (Phase 1 MVP)**
```
프리셋: DOE Reference Buildings 기반
- Large Office (46,320 m², 12F, VAV+Chiller)
- Medium Office (4,982 m², 3F, VRF)
- Small Office (511 m², 1F, PSZ-HP)
- Standalone Retail (2,294 m², 1F, PSZ-AC)
- Primary School (6,871 m², 1F, VAV)
- Hospital (22,422 m², 5F, Chiller+VAV)
- Hotel (Phase 4)
- Apartment (Phase 4)
사용자가 파라미터 조정 (층수, 면적, 창호비 등)
```

#### 4.1.2 3D 모델 생성 파이프라인 (확정: 하이브리드)

```
입력 (자연어/사진/파라미터/템플릿)
        │
        ▼
┌─────────────────────┐
│ Building Parameter   │  ← 중간 표현 (JSON Schema)
│ Schema (BPS)        │     건물의 모든 물리적 특성을 구조화
└─────────────────────┘
        │
        ├─── 단순 형상 (80%) ──→ Three.js 프론트엔드
        │    (템플릿, 파라메트릭)    즉시 렌더링, 서버 불필요
        │
        ├─── 복잡 형상 (15%) ──→ LLM → Three.js 파라미터 변환
        │    (자연어 복잡 요청)      실패 시 Blender 폴백
        │
        └─── 고급 (5%) ────────→ Blender headless (서버)
             (사진/도면/IFC)        GPU 온디맨드, 비용 최소화
        │
        ▼
┌─────────────────────┐
│ 출력                 │
│ · glTF/glb (웹 뷰어)│
│ · IDF (EnergyPlus)  │
│ · BPS (수정용 JSON) │
└─────────────────────┘
```

**비용 최적화**:
- Three.js 주력 (80%): 서버 비용 $0, 즉시 응답
- Blender 보조 (5%): GPU 온디맨드, 월 $30~80 (실사용 기준)
- 총 3D 관련 비용: 기존 Blender 전용 대비 **~90% 절감**

### 4.2 모듈 ② — 건물 설정 (Building Configuration)

단계별 위저드 형태로 사용자가 시뮬레이션 조건을 설정.

#### 4.2.1 설정 단계

```
Step 1: 위치 & 기후
├── 지도에서 위치 선택 (또는 자동 감지)
├── EPW 파일 자동 매칭 (한국 10도시)
├── 기후대 표시 (ASHRAE Climate Zone)
└── 설계 외기 조건 표시

Step 2: 운영 조건
├── 용도별 프리셋 (오피스, 학교, 병원 등)
├── 근무일/휴일 캘린더 (한국 공휴일 자동 반영)
├── 운영 시간 (시작/종료)
├── 토요일 운영 여부
└── 재실 밀도

Step 3: HVAC 설비
├── 설비 유형 추천 (건물 규모 기반)
│   ├── Large: VAV + Chiller + Boiler
│   ├── Medium: VRF
│   └── Small: PSZ-HP / PSZ-AC
├── 설비 용량 (autosize 또는 수동)
├── COP/효율 설정
└── 보조 설비 (환기, 가습 등)

Step 4: 설정온도
├── 프리셋: ASHRAE 55 기반 권장값
├── 난방 설정온도 (재실/비재실)
├── 냉방 설정온도 (재실/비재실)
└── PMV 목표값 (0.5 / 0.7 선택)

Step 5: 검토 & 확인
├── 전체 설정 요약 카드
├── 경고/권고 표시 (비합리적 설정 감지)
├── "시뮬레이션 시작" 버튼
└── 예상 소요 시간 표시
```

#### 4.2.2 스마트 기본값

- 건물 유형 + 지역 기반으로 최적 기본값 자동 설정
- P1(비전문가): 대부분 기본값 → 빠른 진행 (3클릭 완료 가능)
- P3(전문가): 모든 값 커스터마이징

### 4.3 모듈 ③ — Baseline 시뮬레이션 (Simulation Engine)

#### 4.3.1 BPS → IDF 변환

```
BPS (JSON)
    │
    ▼
┌─────────────────────┐
│ IDF Generator       │
│ (Python)            │
│                     │
│ 1. 기하학 변환      │  ← Zone, Surface, Window 객체 생성
│ 2. 재료/구조 적용   │  ← 외피 U-value → Construction 매핑
│ 3. HVAC 템플릿 적용 │  ← 건물유형별 HVAC 시스템 자동 구성
│ 4. 스케줄 생성      │  ← 운영조건 → Schedule:Compact
│ 5. 출력 변수 설정   │  ← 표준 미터 + 커스텀 출력
│ 6. 검증             │  ← 물리적 타당성 + 참조 무결성
└─────────────────────┘
    │
    ▼
model.idf + metadata.json (SHA256)
```

#### 4.3.2 시뮬레이션 실행 (확정: Cloud Run + K8s Jobs)

```
┌─────────────────────────────────────────────┐
│ 시뮬레이션 실행 아키텍처                      │
│                                             │
│ 단일 Baseline (1~5분):                      │
│ → GCP Cloud Run 컨테이너                    │
│ → EnergyPlus 24.1 Docker 이미지             │
│ → 자동 스케일 0→N (유휴 시 비용 $0)         │
│ → 최대 실행 시간: 60분                      │
│                                             │
│ M0~M8 배치 (9개 병렬, 5~15분):              │
│ → GKE Kubernetes Jobs                       │
│ → 9개 Pod 동시 실행 (1 Pod = 1 전략)        │
│ → 완료 후 자동 정리                         │
│                                             │
│ 진행 상황 전달:                              │
│ → Redis Pub/Sub → WebSocket → 프론트엔드    │
│ → "M3 시뮬레이션 완료 (4/9)" 실시간 표시    │
│                                             │
│ 비용 (Cloud Run 기준):                      │
│ · 시뮬레이션 1회: ~$0.01                    │
│ · M0~M8 배치: ~$0.09                       │
│ · 월 1,000 배치: ~$90/월                   │
└─────────────────────────────────────────────┘
```

### 4.4 모듈 ④ — EMS 전략 비교 (Strategy Comparison)

#### 4.4.1 자동 EMS 적용

Baseline 완료 후, 사용자에게 질문:
```
"에너지 절감 전략을 자동으로 비교하시겠습니까?"
  [전체 비교 (M0~M8)]  [선택 비교]  [건너뛰기]
```

**전체 비교 선택 시**:
```
BASELINE 결과
    │
    ▼
┌──────────────────────────────┐
│ 공정 조건 자동 적용           │
│                              │
│ - 설비 크기 고정 (Baseline)  │
│ - 동일 기간/기후/타임스텝    │
│ - 동일 재실 스케줄           │
│ - 동일 Setback 온도          │
│ - DataCenter 부하 고정       │
└──────────────────────────────┘
    │
    ▼
M0~M8 IDF 9개 자동 생성 → 9 Pod 병렬 시뮬레이션
```

#### 4.4.2 EMS 전략 설명 (사용자 UI)

| 전략 | 사용자용 이름 | 한 줄 설명 |
|------|-------------|-----------|
| M0 | 야간 정지 | 퇴근 후 HVAC 자동 정지 |
| M1 | 스마트 시작 | 기온에 따라 출근 전 예열/예냉 |
| M2 | 외기 냉방 | 외기가 시원할 때 자연 환기 활용 |
| M3 | 냉동기 단계제어 | 부하에 맞게 냉동기 대수 조절 |
| M4 | 쾌적 최적화 (보통) | PMV 0.5 기준 설정온도 자동 조정 |
| M5 | 쾌적 최적화 (절약) | PMV 0.7 기준 더 넓은 허용 범위 |
| M6 | 설비 통합제어 | M2+M3 복합 적용 |
| M7 | 통합+쾌적(보통) | M6+M4 최적 조합 |
| M8 | 통합+쾌적(절약) | M6+M5 최대 절감 |

#### 4.4.3 건물 유형별 전략 적용 가능성 (자동 필터링)

시스템이 건물 HVAC 유형에 따라 적용 불가 전략을 자동 비활성화:
```
Large Office (VAV+Chiller): M0~M8 전체 적용 가능
Medium Office (VRF):        M0, M4, M5만 적용 (M2, M3, M6~M8 비활성)
Small Office (PSZ-HP):      M0, M2, M4, M5 적용 (M3 비활성 → M6=M2)
Standalone Retail (PSZ-AC): M0, M2, M4, M5 적용 (M3 비활성 → M6=M2)
Primary School (VAV):       M0, M2, M4, M5 적용 (M3 스킵 → M6=M2)
Hospital (Chiller+VAV):     M0~M8 전체 적용 가능
```
